## 决策：工作模式设计 — debug 重命名为 edit

- 日期：2026-02-24
- 标签：模式设计, 工作流, 命名
- 状态：已采纳

**摘要**：将 `debug` 模式重命名为 `edit`，调试场景改为 `explore`（诊断）→ `edit`（修复）两步流程。

---

### 背景

原设计有四个模式：explore / debug / plan / build。
`debug` 模式的实际约束是"≤2个文件的写操作"，与"调试"认知场景并无强绑定——用它实施小型功能改动同样合理，但名字产生了语义误导。

### 调研发现

- Cursor / Copilot 均使用 Ask / **Edit** / Agent 三模式，无 debug 专属模式
- Aider 使用 /ask / Architect / **Editor** 分层
- 主流工具均以"写权限范围"而非"认知任务类型"来区分模式
- 调试的诊断阶段（只读）与 explore 完全重合，无需独立模式

### 决定

- `debug` → `edit`（小范围写模式，≤2文件，无需 Spec）
- 调试场景工作流：`/dm:explore`（定位根因）→ `/dm:edit`（最小修复）
- 不新增第五个模式（diagnose），避免增加用户认知负担

### 被否决的方向

- 新增 `diagnose` 模式（explore 已覆盖，职责重复）
- 保留 `debug` 名称（语义模糊，对实现功能场景不友好）
